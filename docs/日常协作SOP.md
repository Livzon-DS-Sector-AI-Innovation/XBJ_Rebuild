# 日常协作 SOP（每天都看这份）

> **这份给谁**：本仓库所有人，尤其零经验、靠 AI 辅助开发的同学。
> **什么时候用**：日常改 bug、加小功能、改页面——**90% 的日常工作**用这份就够。
> **不要用这份的情况**：从别的小组**搬整个模块**、大重构、清密钥 → 那是重活，看 [模块移植与合并SOP.md](模块移植与合并SOP.md)。
> **核心心法**：你不用背 git 命令，**不会就问 Claude**，但每一步要**确认它做完了**。

---

## 一、三句话原则（记住这三句就行）

```
1. 干活前：先拉最新的
2. 干活时：在自己的分支上干（别动 main）
3. 干完后：提一个 PR，让队友看一眼再合
```

就这三句。下面是对应的具体操作 + "不会就对 Claude 说的话"。

---

## 二、每天的固定动作

### 开工（每天第一件事）
```powershell
git status                              # 先看工作区干不干净
# 如果有上次没提交的改动: 先决定提交还是 git stash 暂存, 别带进新分支
git checkout main
git pull
git checkout -b 类型/描述-你的名字          # 例: fix/login-button-czl
```
> 分支名格式见下方"分支命名规则"。**只用小写英文 + 短横线**，别用中文。
> 不会就对 Claude 说：**"先看下工作区干净吗，然后从最新 main 给我开个分支叫 fix/login-button-czl"**

### 干活中（随时存档）
```powershell
git status                              # 看改了哪些文件
git diff                                # 看具体改了啥, 删掉调试 print / 注释掉的废代码
# 提交前先本地验证(改哪端跑哪端), 别等 CI 红了再修:
uv run pytest -q                        # 后端: 跑相关测试
pnpm tsc --noEmit                       # 前端: 类型检查
# 确认没崩, 再提交(只加你这次改的文件, 别无脑 git add .):
git add 你改的文件
git commit -m "fix: 修好了登录按钮点击没反应"
```
> 不会就对 Claude 说：**"帮我看改了哪些文件、有没有不该提交的，跑一下相关测试，然后只提交登录相关的改动，说明是修好登录按钮"**
> ⚠️ 提交信息要有前缀 + 说清做了啥（见第三节）。一次只提交一件事。

### 收工（或功能做完）
```powershell
git pull origin main                    # 先同步最新 main(防止落后, 否则 PR 会卡)
# 若有冲突, 对 Claude 说"帮我解决冲突并解释为什么冲突"
git push origin 你的分支名
```
然后去 GitHub 页面点 **Create Pull Request**。
> 不会就对 Claude 说：**"帮我同步最新 main，解决冲突，推上去并准备好提 PR"**

### 分支命名规则（自己建、让 Claude 建都照这个）

格式：**`<类型>/<简短英文描述>-<你的名字或代号>`**

```
feat/employee-export-czl     新功能：员工导出
fix/login-button-li            修 bug：登录按钮
docs/add-sop-czl             文档：加 SOP
```

类型前缀（选最贴切的一个）：

| 前缀 | 用在 | 前缀 | 用在 |
|---|---|---|---|
| `feat/` | 新功能 | `docs/` | 文档 |
| `fix/` | 修 bug | `refactor/` | 重构 |
| `chore/` | 杂项/配置/依赖 | `freeze/` | 冻结别人代码（移植场景，见移植 SOP） |

规则：
- **只用小写英文 + 数字 + 短横线（-）+ 斜杠（/）**，不用中文、空格、下划线（中文分支名在某些工具会编码出错）。
- 描述 **2-4 个词**，一眼看出干啥；**名字/代号放最后**，方便 @ 到人。
- **一个分支只做一件事**（对应一个 PR）；分支**存活别超过 3 天**，尽快合或删。
- **永远不在 `main` 上直接改**——`main` 是受保护分支。

> 让 Claude 建分支时直接说：**"从最新 main 开个分支，按命名规则叫 `feat/xxx-我的名字`"**，它会照这个格式建。

## 三、提交信息（commit）怎么写

格式：**`前缀: 中文说清做了啥`**

| 前缀 | 用在 | 例子 |
|---|---|---|
| `feat:` | 加新功能 | `feat: 增加员工导出 Excel 按钮` |
| `fix:` | 修 bug | `fix: 修复离职台账日期显示错乱` |
| `chore:` | 杂项（配置、依赖、格式） | `chore: 更新 .gitignore` |
| `docs:` | 改文档 | `docs: 补充 README 启动步骤` |

铁律：
- **一个提交只做一件事**。改了功能又顺手修了别的 bug → 分两个提交。
- **文件头部注释**：让 Claude 改文件时，它会**自动按 `CLAUDE.md` 规范生成/更新**头部注释（业务+依赖）——你只需**确认业务归属写对了**（这文件到底属于哪个模块、干啥的）。
- 不要提交 `print`、注释掉的废代码、临时调试（提交前 `git diff` 扫一眼，或让 Claude 帮你查）。

## 四、提 PR 和合并

1. push 后去 GitHub 点 **Create Pull Request**，base 选 `main`。
2. PR 标题写清做了啥（和 commit 一个风格）。
3. **合并前确认你的分支没有垃圾 commit**（像 `wip`、`测试一下`、`再改改`）。有的话让 Claude 帮你整理成一两个有意义的提交再 push——否则它们会随 merge 进 main，弄脏历史。
4. **等两件事都满足才能合**：
   - ✅ CI 两个检查（`backend` + `frontend`）变绿。**两个每次都会跑、都要绿**（CI 没做按目录跳过，别等"另一个"——它一定也在跑）。
   - ✅ 一个**队友** Approve（GitHub 不让自己批自己的）。
5. 合并时选 **"Create a merge commit"**，**不要选 "Squash"**。

> **互批默契**：提了 PR 在群里 @ 一下，队友谁先看到谁批。批的人主要看两点：①有没有动到不该动的模块（扫一眼 diff 的文件列表，只该改与标题相关的）；②有没有明显会搞坏别人东西的地方。CI 已经验过能不能跑，人主要扫方向，30 秒。
> **没把握就别硬批**：如果你看不懂这个 PR 改的模块，在 PR 里 @ 真正懂那块的人来看，不要为了走流程随便点 Approve。

> **只有你一个人在线、PR 卡住怎么办**：这是例外。① 在 PR 评论写明"唯一协作者在线，请求临时 bypass"；② 找仓库 **Owner/管理员**（维护者群里）临时把审核数调 0；③ 合完后**管理员必须立刻调回 1，并在群里通知**"刚临时降级合了 X，已恢复"。别养成习惯——这是应急，不是常态。

## 五、遇到问题怎么办（直接问 Claude）

| 情况 | 对 Claude 说 |
|---|---|
| git 报冲突了 | "git 报冲突了，帮我看怎么解决，先解释为什么冲突" |
| PR 上 CI 红了 | "PR 的 CI 红了，帮我看哪个检查失败、为什么、怎么修" |
| 别人合了 main，我的分支落后 | "main 更新了，帮我把最新 main 同步到我的分支" |
| 不知道改对没 | "帮我验证这个改动，跑一下相关测试和类型检查" |

> 关键：**让 Claude 解释"为什么"，你才学得会**。别只复制答案——这是你从"调包侠"变成"能排障"的唯一路径。
> **问之前先看一眼报错**：把命令行/页面上的**报错原文发给 Claude**——报错本身就是最重要的线索，比"它报错了"有用一百倍。

## 六、铁律（红线，别碰）

- ❌ **永远不要直接 push 到 main**（分支保护会拦，但别试）。
- ❌ **永远不要自己点 Merge 自己没人批的 PR**（要等队友 Approve）。
- ❌ **永远不要 `git push --force` 到 main**（清密钥等破坏性操作是管理员专项，见移植 SOP）。
- ❌ **永远不要把密钥/密码写进代码或 `.env.example`**。需要新增配置时，参照 `.env.example` 已有格式加一行，值**一律填 `YOUR_XXX` 占位**，真值只放进本地 `.env`（已被 gitignore，不会提交）。
- ⚠️ **万一不小心提交了真密钥**：别自己乱试删除。马上找管理员，按 [移植 SOP 第 ⑦ 步](模块移植与合并SOP.md) 从历史里清除 + 重置密钥。
- ✅ **不会就问 Claude，但要它解释，并确认每步做完了**。

---

## 七、一句话总结

```
开工拉最新 → 开自己分支干 → commit 写清楚 → push 提 PR
→ 等 CI 绿 + 队友批 → merge commit 合 → 收工
全程不会就问 Claude，让它解释、你确认。
```
