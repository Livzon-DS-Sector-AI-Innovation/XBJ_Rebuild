# 新人上手指南（第一天照着做）

> **这份给谁**：刚加入、或来自其他小组、零基础靠 AI 辅助开发的新人。
> **目标**：第一天**半天能把项目跑起来**，下午**提出你的第一个 PR**。
> **心法**：不会就问 Claude，让它解释、你确认。报错原文直接发给它。
> 跑通后，日常看 [日常协作SOP.md](日常协作SOP.md)；要搬模块/大重构看 [模块移植与合并SOP.md](模块移植与合并SOP.md)。

## 上午：把环境装好、项目跑起来

### 1. 装工具（一次性）
- **VSCode** + **Claude Code 扩展**（发布者 Anthropic；装完按提示登录 Anthropic 账号；首次打开项目选"信任此文件夹"）。团队统一用这个，别用别的编辑器。
- **Git**、**Docker Desktop**（数据库 Redis/Postgres 跑在容器里）。
- 后端：**uv**（Python 包管理）　前端：**pnpm**（Node 22）。
  > 不会装就把这句发给 Claude：**"指导我在 Windows 上装 uv 和 pnpm，Node 用 22"**
- **配一次 git 身份**（否则提交作者显示 unknown）：
  ```powershell
  git config --global user.name "你的名字"
  git config --global user.email "你的邮箱"
  ```

> ℹ️ Claude Code 会**给你命令并能直接在终端执行**（你确认后运行），每步完成把结果或报错贴给它。

### 2. 拿到代码
```powershell
git clone <仓库地址>
cd XBJ
```
> 不会就对 Claude 说：**"帮我 clone 这个仓库并进入目录"**

### 3. 配环境变量
```powershell
cd dazah-backend
cp .env.example .env       # 复制一份
```
然后**找团队负责人要真实的密钥值**填进 `.env`（飞书 APP_SECRET、MOONSHOT_API_KEY 等）。
> ⚠️ `.env.example` 里是 `YOUR_XXX` 占位符，真值只填进 `.env`。**别把真密钥填回 .env.example**。
> ℹ️ `.env` 已在 `.gitignore` 里，`git add` 不会包含它，**不会泄漏，别担心也别手动删**。
> 不确定哪些必填？把 `.env.example` 内容发给 Claude 问：**"哪些变量本地开发必须填？"** 负责人不在时，也可让 Claude 读 CI 配置推断。

### 4. 跑起来（让 Claude 帮你）
直接对 Claude 说：**"帮我把这个项目在本地跑起来"**。它大致会做：
```powershell
# 0. 先启动数据库/缓存容器(关键!不启后端连不上库)
cd dazah-backend
docker compose up -d            # 起 postgres + redis
docker ps                       # 确认 db、redis 两个容器在跑
# (每次重启电脑后,都要先 docker compose up -d 再启项目)

# 1. 后端
uv sync --extra dev                                    # 装依赖(含 dev 工具)
uv run alembic upgrade head                            # 建表
uv run uvicorn app.main:app --reload --port 8000       # 起后端

# 2. 前端(新开一个终端)
cd dazah-frontend; pnpm install; pnpm dev              # 起前端
```
> ⚠️ **若 `alembic upgrade head` 报错**：本项目迁移链有已知不齐（见移植 SOP 遗留债）。备选方案是用 ORM 直接建表：`uv run python scripts/ci_create_tables.py`，或把报错发给 Claude 让它诊断。**别自己改迁移文件。**

跑通的标志：
- 后端 `http://localhost:8000/docs` 能打开（API 文档）
- 前端 `http://localhost:3000` 能打开（页面）

> 卡住了把**报错原文**发给 Claude，它帮你查（常见坑见各 SOP 的踩坑清单）。
> 依赖安装/拉镜像耗时看网络，慢是正常的，等的时候可以去读下面的文档，别干等。

## 下午：提出你的第一个 PR（走一遍完整流程）

### 1. 认领一个最简单的任务
找负责人要一个小到不能再小的活（改个文案、修个小样式、加个字段注释都行）。**第一个 PR 重点是跑通流程，不是干大事。**
> 负责人暂时不在、没任务？**自己造一个**：修个注释错别字、给某函数补一行中文注释、改个无关痛痒的前端文案（如"确定"→"确认"）。重点是走通 PR 全流程。

### 2. 照 [日常协作SOP.md](日常协作SOP.md) 走一遍
```powershell
git status                                   # 看工作区干净
git checkout main; git pull                  # 拉最新
git checkout -b fix/xxx-你的名字              # 开分支(类型/描述-名字)
# ...改代码...
git diff                                     # 看改了啥, 删掉调试 print/废代码

# 提交前本地验证(确认 docker 容器在跑, 否则 pytest 连不上库会假报错):
docker ps                                    # 确认 db/redis 在
cd dazah-backend; uv run pytest -q           # 后端改动: 在后端目录跑
cd dazah-frontend; pnpm tsc --noEmit         # 前端改动: 在前端目录跑

git add 你改的文件; git commit -m "fix: 说清做了啥"
git pull origin main                         # 同步最新(防落后)
git push origin 你的分支名
```
> ⚠️ `pytest`/`tsc` 必须**在对应子目录**（`dazah-backend` / `dazah-frontend`）里跑，在根目录跑会报"找不到模块"。
> ⚠️ `git pull origin main` 若提示**冲突（conflict）**：别自己乱改，把报错 + `git status` 发给 Claude，让它一步步带你解决。
> 全程不会就问 Claude，每句话都可以是"帮我 xxx，并解释你做了什么"。

push 后去 GitHub 页面点 **Create Pull Request**，base 选 `main`，标题写清做了啥。
> ⚠️ **万一不小心提交了真密钥**：别自己乱试删除，马上找管理员，按 [移植 SOP 第 ⑦ 步](模块移植与合并SOP.md) 从历史清除 + 重置密钥。

### 3. 等 CI + 队友审核 → 合并
- CI 两个检查（`backend`、`frontend`）变绿。**两个每次都会跑、都要绿**（CI 没按目录跳过，别等"另一个"——它一定也在跑；若一直灰色/等待，群里问是不是配置问题）。
- 找队友 Approve 一个（你自己不能批自己的）。
- 合并选 **"Create a merge commit"**，不要 squash。

**恭喜，你的第一个 PR 进 main 了。** 流程跑通，以后就是重复这套。

## 你的成长路线（别只当"调包侠"）

| 时间 | 目标 |
|---|---|
| 第 1 周 | 能照流程独立提 PR、走完 CI + 审核 |
| 第 1 个月 | 能看懂 AI 写的代码，遇事会先问"为什么这么写" |
| 第 3 个月 | AI 辅助下能独立排障、定位问题，接近中级水平 |

关键习惯（决定你能不能成长）：
- 每次让 AI 解决问题，**追问"为什么"**，把学到的记下来。
- 遇报错**先自己读一眼**，再连原文发给 Claude。
- 别怕问，但别只复制答案——**理解了才是你的**。
