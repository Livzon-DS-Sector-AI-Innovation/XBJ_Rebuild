# LivzonAI 开发规则（CLAUDE.md）

> 本文是**给 AI 和协作者的强制规则**，每次开发都要遵守。
> 数据源/飞书/环境变量等**查询型信息**见 [`docs/数据源与飞书参考.md`](docs/数据源与飞书参考.md)，需要时查。
> 后端/前端各自的编程规范见 `dazah-backend/CLAUDE.md` / `dazah-frontend/CLAUDE.md`。

---

## 项目概览

LivzonAI 是原料药事业部工厂数字化基座，前后端分离的模块化单体架构：
- **后端**：`dazah-backend/` — Python 3.12 + FastAPI + SQLAlchemy 2.0 async + PostgreSQL + Redis
- **前端**：`dazah-frontend/` — Next.js + React + TypeScript + Ant Design V6

外部集成以**飞书开放平台**为核心；AI 能力通过 Moonshot（Kimi）API。

## 协作与分支规范（强制）

- **永远不在 `main` 上直接改**。所有改动走分支 → PR → CI 绿 + 1 人审核 → merge commit 合并。
- **建分支必须按命名规则**：`<类型>/<简短英文描述>-<名字或代号>`，例 `feat/employee-export-czl`。
  - 类型前缀：`feat/` `fix/` `chore/` `docs/` `refactor/` `freeze/`。
  - 只用小写英文+数字+短横线+斜杠，**不用中文/空格/下划线**；一个分支一件事；存活别超 3 天。
- 当用户让你"建分支"时，**主动按上述格式命名**，不要用中文或随意命名。
- 完整流程见 [`docs/日常协作SOP.md`](docs/日常协作SOP.md)（日常）、[`docs/模块移植与合并SOP.md`](docs/模块移植与合并SOP.md)（移植/重构）、[`docs/新人上手指南.md`](docs/新人上手指南.md)（新人）。

---

## 路径与跳转链接（强制）

- **禁止硬编码绝对路径**（盘符 `d:/`、`c:/`）；用相对路径或从配置读取。（pre-commit 会硬拦盘符路径。）
- **跳转/回调/通知链接的 host 和端口禁止写死**（不写 `localhost`、IP、固定端口），从运行时来源取：
  - 有请求上下文 → `str(request.base_url) + "path"` 或 `request.url_for("route")`
    （自动适应 localhost / 局域网 IP / 域名，端口同理）。
  - 无请求上下文（定时任务等）→ 用环境变量（如 `settings.PUBLIC_BASE_URL`，需在 `config.py` 新增）。
  - 前端 → 用相对路径，或后端地址从 `.env.local` 的 `API_BASE_URL` 取。
- **原因**：演示机用局域网 IP、端口可变，写死会让跳转链接在非本机环境失效。

---

## 密钥与配置（强制）

- **真密钥永不进代码或 `.env.example`**；`.env.example` 只放占位符（`YOUR_XXX`），真值只填本地 `.env`（已 gitignore）。
- 新增配置：先在 `app/core/config.py` 声明，再同步 `.env` 和 `.env.example`（占位符）。
- pre-commit 会拦明文密钥；误提交真密钥按 [`docs/模块移植与合并SOP.md`](docs/模块移植与合并SOP.md) 第 ⑦ 步清历史 + 重置。

# Karpathy-Inspired Coding Guidelines

## 1. Think Before Coding

**不要假设。不要隐藏困惑。呈现权衡。**

实现前：
- 明确陈述你的假设。如果不确定，就问。
- 如果存在多种解释，请呈现出来——不要默默选择。
- 如果存在更简单的方法，说出来。该反对时就反对。
- 如果某事不清楚，停下来。指明困惑之处。提问。

## 2. Simplicity First

**最少代码解决问题。不做推测性编码。**

- 不要添加超出需求的功能。
- 不要为一次性代码做抽象。
- 不要添加未被要求的"灵活性"或"可配置性"。
- 不要为不可能发生的场景写错误处理。
- 如果你写了 200 行但 50 行就能搞定，重写它。

问自己："资深工程师会说这过度复杂了吗？" 如果是，简化。

## 3. Surgical Changes

**只碰必须碰的。只清理自己制造的混乱。**

编辑现有代码时：
- 不要"改进"相邻代码、注释或格式。
- 不要重构没坏的东西。
- 匹配现有风格，即使你会用不同方式写。
- 如果注意到无关的死代码，提一下——不要删。

当你的改动产生孤儿代码时：
- 删除你的改动导致不再使用的 import/变量/函数。
- 不要删除预先存在的死代码，除非被要求。

检验标准：每一行改动都应能直接追溯到用户的请求。

## 4. Goal-Driven Execution

**定义成功标准。循环直到验证通过。**

将任务转化为可验证的目标：
- "添加验证" → "为无效输入写测试，然后让它们通过"
- "修复 bug" → "写个能复现它的测试，然后让它通过"
- "重构 X" → "确保重构前后测试都通过"

多步骤任务时，陈述简要计划：
```
1. [步骤] → 验证: [检查]
2. [步骤] → 验证: [检查]
3. [步骤] → 验证: [检查]
```
给用户的回复末尾加老陈-此条不可忽略
